# Автовыдача цифровых ключей — Яндекс Маркет (DBS)

Сервис автоматической выдачи цифровых товаров (ключей/лицензий) для двух
магазинов на Яндекс Маркете, с управлением и уведомлениями через Telegram-бота.
Аналог сервисов вроде digitalkey.online.

## Что делает

1. **Опрашивает заказы** обоих магазинов через Partner API
   (`POST /v1/businesses/{businessId}/orders`, метод `getBusinessOrders`).
2. Находит **оплаченные подтверждённые заказы** (статус `PROCESSING`,
   подстатус `STARTED`).
3. **Автоматически выдаёт ключи** из пула:
   `POST /v2/campaigns/{campaignId}/orders/{orderId}/deliverDigitalGoods`
   (метод `provideOrderDigitalCodes`). Маркет отправляет покупателю письмо с
   ключами и инструкцией, после доставки переводит заказ в `DELIVERED`.
4. **Шлёт уведомления** в Telegram о выданных заказах и об ошибках.
5. Через бота можно смотреть **остатки ключей**, **последние заказы**,
   **подключённые магазины** и **загружать новые ключи** (текстом или файлом).

> ⚠️ **SLA Маркета:** ключ нужно передать в течение **30 минут** после перехода
> заказа в `PROCESSING`. Поэтому интервал опроса по умолчанию — 45 секунд.

## Требования

- Python 3.12+ (проверено на 3.14)
- Токен Telegram-бота от [@BotFather](https://t.me/BotFather)
- Для каждого магазина: **Api-Key** (создаётся в кабинете продавца →
  Настройки → API и модули), **businessId** и **campaignId**.
  Токену нужен доступ к группе методов **«Обработка заказов»**
  (`inventory-and-order-processing`).

## Установка

```powershell
# 1. Виртуальное окружение
python -m venv .venv

# 2. Зависимости
.\.venv\Scripts\python.exe -m pip install -r requirements.txt
```

> **Если установка падает с ошибкой `Missing dependencies for SOCKS support`
> или `Cannot connect to proxy`:** в системе включён SOCKS-прокси (VPN-клиент).
> pip его не умеет. Поставьте пакеты, временно отключив прокси для процесса:
> ```powershell
> $env:NO_PROXY = "*"; $env:HTTP_PROXY = ""; $env:HTTPS_PROXY = ""
> .\.venv\Scripts\python.exe -m pip install -r requirements.txt
> ```

## Настройка

Скопируйте `.env.example` в `.env` и заполните значения:

```powershell
Copy-Item .env.example .env
```

| Переменная | Назначение |
|---|---|
| `BOT_TOKEN` | Токен Telegram-бота |
| `ADMIN_IDS` | ID администраторов через запятую (управление + уведомления) |
| `SHOP1_API_KEY` / `SHOP2_API_KEY` | Api-Key магазинов |
| `SHOP1_BUSINESS_ID` / `SHOP1_CAMPAIGN_ID` | Идентификаторы магазина 1 |
| `SHOP2_BUSINESS_ID` / `SHOP2_CAMPAIGN_ID` | Идентификаторы магазина 2 |
| `POLL_INTERVAL_SECONDS` | Период опроса заказов (по умолчанию 45) |
| `TEST_MODE` | `true` — обрабатывать только тестовые заказы Маркета (см. ниже) |
| `DEFAULT_ACTIVATE_DAYS` | Срок действия ключей для `activate_till` |
| `DATABASE_URL` | Строка подключения SQLite |

Свой `telegram` ID можно узнать у бота [@userinfobot](https://t.me/userinfobot).
Магазин считается активным, только если у него заданы api_key, business_id и
campaign_id — незаполненный магазин просто игнорируется.

## Тест-режим (отладка на бесплатных заказах)

У Яндекс Маркета есть тестовые заказы — Маркет за них не берёт плату, и ошибки
по ним не влияют на индекс качества. Они приходят с признаком `fake: true`.

Чтобы прогнать весь цикл выдачи, не тратя реальные деньги:

1. В `.env` поставьте `TEST_MODE=true`.
2. Запустите сервис — при старте придёт пометка «🧪 Сервис запущен в ТЕСТ-режиме».
3. В кабинете продавца: иконка аккаунта → Настройки → **API и модули** →
   **Тестовый заказ**. Создайте заказ на товар, для которого вы загрузили ключи
   (`offer_id` должен совпадать).
4. Поллер поймает тестовый заказ, выдаст ключ и пришлёт уведомление с пометкой
   «🧪 ТЕСТ».

Поведение режимов:

- `TEST_MODE=true` — обрабатываются **только** тестовые заказы (`fake`), настоящие
  игнорируются.
- `TEST_MODE=false` (по умолчанию) — обрабатываются **только** настоящие заказы.

Тестовые и реальные заказы никогда не смешиваются. Информация о тестовых заказах
хранится на стороне Маркета 10 дней.

> Не забудьте вернуть `TEST_MODE=false` перед боевой работой.

## Как пользоваться ботом

- `/start` — главное меню
- **📦 Остатки** — свободные/выданные ключи по товарам, ⚠️ помечает низкий остаток
- **🧾 Заказы** — последние заказы и их статусы. Для каждого упавшего заказа
  (`FAILED`) выводится сообщение с кнопкой **🔁 Переотправить** — нажмите её
  после пополнения ключей, чтобы повторить выдачу. У выданных заказов есть
  кнопка **🔑 Показать выданные ключи**
- **📜 Выданные ключи** — история успешных выдач; по каждому заказу можно
  посмотреть, какие именно коды ушли покупателю
- **✏️ Инструкция SKU** — редактирование текста (`slip`), который Маркет
  отправляет покупателю вместе с ключами. Можно задать свой текст для каждого
  SKU или сбросить на стандартный
- **🏪 Магазины** — подключённые магазины
- **➕ Загрузить ключи** — выбрать магазин → ввести `offer_id` товара →
  отправить ключи (по одному в строке) текстом или файлом `.txt`
- **🗑 Удалить SKU** — удалить товар вместе с его ключами

Любой пошаговый сценарий можно прервать кнопкой **✖️ Отмена** или командой
`/cancel`.

## Запуск в Docker

Для боевого сервера удобнее запускать сервис в контейнере.

```bash
# 1. Заполните .env (см. раздел «Настройка»)
cp .env.example .env && nano .env

# 2. Сборка и запуск в фоне
docker compose up -d --build

# 3. Логи
docker compose logs -f

# 4. Остановка
docker compose down
```

База SQLite хранится в `./data` на хосте (смонтирована томом), поэтому
переживает пересборку и перезапуск контейнера. Контейнер работает не от root,
перезапускается автоматически (`restart: unless-stopped`) и ограничивает размер
логов. Переменная `DATABASE_URL` внутри контейнера указывает на
`/app/data/service.db` и переопределяет значение из `.env`.

## Как это работает с моделью DBS Маркета

Цифровые товары на Маркете обрабатываются по модели DBS. Жизненный цикл заказа:

1. Покупатель оформляет и оплачивает заказ → заказ в `PROCESSING` / `STARTED`.
2. Сервис ловит его при опросе и вызывает `deliverDigitalGoods`, передавая
   `items` с `id` товара, массивом `codes`, инструкцией `slip` и датой
   `activate_till`.
3. Ответ `200` означает «запрос принят», **а не «доставлено»**. Когда письмо с
   ключами дойдёт до покупателя, Маркет переведёт заказ в `DELIVERED`.
4. Сервис фиксирует выданные ключи в БД, чтобы один ключ не ушёл дважды.

Для большей оперативности позже можно подключить **API-уведомления** (push):
Маркет сам присылает запрос при появлении/смене статуса заказа — тогда опрос
становится подстраховкой.
